# Wire Protocol

The `server` binary exposes a raw binary HTTP API — no JSON anywhere on the
hot path. This is the single source of truth for the wire format; it is
generated by reading `src/server.rs`, `src/board.rs`, `src/game.rs` and
`src/moves.rs` directly, not copied from an external spec.

All requests/responses use `Content-Type: application/octet-stream`.
CORS is wide open (`Any`/`Any`/`Any`) — the engine has no auth of its own and
is expected to sit behind a trusted caller (e.g. keres-platform), not be
exposed directly to end users.

`PORT` env var selects the listen port (default `3000`).

## Board dimensions

- `BOARD_DIMENSION = 9` (9×9 board)
- `BOARD_SIZE = 81` (`BOARD_DIMENSION²`)
- A `Position` is `y * 9 + x`, encoded as a single `u8` in `0..=80`.

## Piece encoding — 1 byte per square

```
bit:    7  6  5  4  3  2  1  0
        0  C  U  U  U  L  L  L
```

- `0` = empty square.
- `C` (bit 6): `1` = White, `0` = Black.
- King: special-cased as `C 111 000` (bottom=King, no top — the king can
  never be stacked).
- Otherwise: `LLL` = bottom piece type (3-bit `PieceType` discriminant),
  `UUU` = top piece type of the stack, or `000` if unstacked (a single
  piece is never confused with an "unstacked King" because King always
  uses its dedicated `111 000` pattern).

`PieceType` discriminants (`src/board.rs`): `Soldier=1, Bishop=2, Rook=3,
Paladin=4, Guard=5, Knight=6, Ballista=7` (`King` has no ordinary
discriminant — it only ever appears via the special byte above).

## Game (board + turn state) — 83 bytes

`Game::to_binary()` / `Game::from_binary()`:

| Bytes | Content                                                                                |
|-------|------------------------------------------------------------------------------------------|
| 0–80  | 81 board squares, one byte each (see piece encoding above)                               |
| 81    | Flags byte: `0x80` white-to-move, `0x40` game-over, `0x20` white-wins, `0x10` draw       |
| 82    | `moves_without_capture` counter (`u8`, drives the 50-move draw rule)                      |

This is the `BOARD_SIZE + 2` = 83-byte format sent/received by `/new`,
`/moves`, `/play`, `/replay-moves`, `/engine-move-board`.

## Move — 2 bytes (`u16`, little-endian)

Two different bit layouts share the same 16-bit slot depending on direction:

**Server → client** (`/moves` response — a `PotentialMove`, `moves.rs`):

```
bit 15: force_unstack   bit 14: unstackable   bits 7-13: to (7 bits)   bits 0-6: from (7 bits)
```

- `unstackable`: this move can legally be played either stacked or
  unstacked — the client must offer the choice (the platform's UX shows a
  confirmation dialog, defaulting to "full stack").
- `force_unstack`: the destination forces the top piece to separate;
  only the unstacked variant is legal.

**Client → server** (`/play`, `/engine-move-*` request/response — a `Move`,
`moves.rs`):

```
bit 14: unstack   bits 7-13: to (7 bits)   bits 0-6: from (7 bits)
```

To play a move: take a `PotentialMove` from `/moves`, decide `unstack`
(`true` only if `unstackable` or `force_unstack` was set), then re-encode as
a plain `Move` with bit 14 = that boolean (bit 15 unused/zero) before
sending it to `/play`.

## Endpoints

| Method | Path                 | Request body                                          | Response body                                   |
|--------|----------------------|--------------------------------------------------------|--------------------------------------------------|
| GET    | `/new`               | —                                                        | 83 bytes: a fresh `Game`                          |
| POST   | `/moves`             | 83 bytes: `Game`                                         | `N × 2` bytes: legal `PotentialMove`s (LE `u16`s) |
| POST   | `/play`              | 85 bytes: 83-byte `Game` + 2-byte `Move`                 | 83 bytes: the resulting `Game`                    |
| POST   | `/replay-moves`      | `N × 2` bytes: `Move` list from the initial position     | 83 bytes: the resulting `Game`                    |
| POST   | `/engine-move-board` | 83 bytes: `Game`                                          | 2 bytes: the engine's chosen `Move`                |
| POST   | `/engine-move-game`  | `N × 2` bytes: full `Move` history from the initial position | 2 bytes: the engine's chosen `Move`           |

`/engine-move-game` replays the given move list from a fresh `Game`,
reconstructing the position-hash history so the search's loop/repetition
detection sees the same game the caller has been playing (`/engine-move-board`
has no history, so it cannot detect repetition draws). Both run
`root_search` at `MAX_DEPTH` (`src/engine/constants.rs`) and respond
`500` if no legal move exists (i.e. the game is already over).

Malformed/mis-sized payloads get `400`; internal decode failures get `500`.

## AI search

See `src/engine/search/` — Negamax with alpha-beta pruning
(`alpha_beta.rs`, `negamax.rs`), quiescence search past the horizon
(`quiescence.rs`), a transposition table (`tt.rs`), killer-move ordering
(`killer.rs`), and repetition/loop detection fed by the caller-supplied game
history (`loop_detection.rs`). Search is parallelized with
[Rayon](https://github.com/rayon-rs/rayon). `root_search` (`search/mod.rs`)
is the entry point used by both engine-move endpoints and by the `keres`
CLI's `engine-move`/`debug-tree` subcommands.
